-- T Design and Implementation of Efficient Message Scheduling for Controller Area Network . -- A AbstractThe Controller Area Network (Controller Area Network ) is being widely used in real-time control applications such as automobiles , aircraft , and automated factories . In this paper , we present the mixed traffic scheduler (mixed traffic scheduler ) for Controller Area Network , which provides higher schedulability than fixed-priority schemes like deadline-monotonic (deadline monotonic ) while incurring less overhead than dynamic earliest-deadline (earliest-deadline 1 ) scheduling . We also describe how mixed traffic scheduler can be implemented on existing Controller Area Network network adapters such as Motorola 's TouCAN . In previous work [ 1 ] , [ 2 ] , We had shown mixed traffic scheduler to be far superior to deadline monotonic in schedulability performance . In this paper , we present implementation overhead measurements showing that processing needed to support mixed traffic scheduler consumes only about 5 to 6 percent of CPU time . Considering its schedulability advantage , this makes mixed traffic scheduler ideal for use in control applications . -- B Introduction Distributed real-time systems are being used increasingly in control applications such as in automobiles , aircraft , robotics , and process control . real-time systems consist of multiple computational nodes , sensors , and actuators interconnected by a LAN [ 3 ] . Controller Area Network (Controller Area Network ) The Controller Area Network specification defines the physical and data link layers (layers 1 and 2 in the ISO/OSI reference model ) . Each Controller Area Network frame has seven fields as shown in Figure 1 , but we are concerned only with the data length (data length ) and the identifier (identifier ) fields . The data length field is 4 bits wide and specifies the number of data bytes in the data field , from 0 to 8 . The data length field can be of two lengths : the standard format is 11-bits , whereas the extended format is 29-bits . The data length field controls both bus arbitration and message addressing , but we are interested only in the former which is described next . Controller Area Network makes use of a wired-OR (or wired-AND ) bus to connect all the nodes (in the rest of The paper we assume a wired-OR bus ) . When a processor has to send a message a processor first calculates the message identifier which may be based on the priority of the message . The identifier for each message must be unique . Processors pass Processors messages and associated IDs to IDs bus interface chips . their bus interface chips wait till the bus is idle , then write The identifier for each message on the bus , one bit at a time , starting with the most significant bit . After writing each bit , each chip waits long enough for signals to propagate along the bus , then each chip reads the bus . If a chip had written a 0 but reads a 1 , a chip means that another node has a message with a higher priority . If so , another node drops out of contention . In the end , there is only one winner and another node can use the bus . This can be thought of as a distributed comparison of the IDs of all the messages on different nodes and the message with The identifier for each message is selected for transmission . 3 Workload Characteristics In control applications , some devices exchange periodic messages (such as motors and drives used in industrial applications ) while others are more event-driven (such as smart sensors ) . Moreover , operators may need status information from various devices , thus generating messages which do not have timing con- straints . So , we classify messages into three broad categories , (1 ) hard-deadline periodic messages , (2 ) hard-deadline sporadic messages , and (3 ) non-real-time (best-effort ) aperiodic messages . A periodic message has multiple invocations , each one period apart (note that whenever we use the term message stream to refer to a periodic , we are referring to all invocations of that periodic ) . Sporadic messages have a minimum interarrival time (MIT ) between invocations , while non-real-time messages are completely aperiodic , but Sporadic messages do not have deadline constraints . Low-Speed vs . High-Speed Real-Time Messages Messages in a real-time control system can have a wide range of deadlines . For example , messages from a controller to a high-speed drive may have deadlines of few hundreds of microseconds . On the other hand , messages from devices such as temperature sensors can have deadlines of a few seconds because the physical property being measured (temperature ) changes very slowly . Thus , we further classify real-time messages into two classes : high-speed and low-speed , depending on the tightness of real-time messages deadlines . As will be clear in Section 4 , the reason for this classification has to do with the number of bits required to represent the deadlines of messages . Note that `` high-speed '' is a relative term - relative to the tightest deadline D 0 in the workload . All messages with the same order of magnitude deadlines as D 0 (or within one order of magnitude difference from D 0 ) can be considered high-speed messages . All others will be low-speed . 4 The Mixed Traffic Scheduler Fixed-priority deadline monotonic (deadline monotonic ) scheduling [ 12 ] can be used for Controller Area Network by setting each message's identifier to each message's identifier unique priority as in [ 8 , 9 ] . However , in general , fixed-priority schemes give lower utilization than other schemes such as non-preemptive earliest-deadline 1 (earliest-deadline 1 ) . This is why several reaserchers have used earliest-deadline 1 for network scheduling [ 15-17 ] . This motivates us to use earliest-deadline 1 to schedule messages on Controller Area Network , meaning that the message identifier must contain the message deadline (actually , the logical inverse of the deadline for a bus ) . But as time progresses , absolute deadline values get larger and larger , and eventually absolute deadline values will overflow the message identifier . This problem can be solved by using some type of a wrap-around scheme (which we present in Section 4.1 ) but even then , puting the deadline in the identifier forces one to use the extended Controller Area Network format with one 29-bit IDs . Compared to the standard Controller Area Network format with 11-bit IDs , this wastes 20-30 % bandwidth , negating any benefit obtained by going from fixed-priority to dynamic-priority scheduling . This makes earliest-deadline 1 impractical for Controller Area Network . First , we give high-speed messages priority over low-speed and non-real-time ones by setting the most significant bit to 1 in The identifier for each message for high-speed messages (Figure 2a ) . This protects high-speed messages from all other types of traffic . If the uniqueness field is to be 5 bits [ 2 ] (allowing high-speed messages ) , and the priority field is 1 bit , then the remaining 5 bits are still not enough to encode the deadlines (relative to the latest start of epoch ) . Our solution is to quantize time into regions and encode deadlines according to which region they fall in . To distinguish messages whose deadlines fall in the same region , Our use the DM-priority of a message as its uniqueness code . This makes mixed traffic scheduler a hierarchical scheduler . At the top level is earliest-deadline 1 : if the deadlines of two messages can be distinguished after quantization , then the one with the earlier deadline has higher priority . Non-preemptive scheduling under release time constraints is NP-hard in the strong sense [ 13 ] . To avoid this problem , we must use an agreement protocol to trigger the identifier update on all nodes . The Controller Area Network clock synchronization algorithm [ 18 ] synchronizes clocks to within 20-s. A simple agreement protocol can be that one node is designated to broadcast a message on the Controller Area Network bus . a message on the Controller Area Network bus will be received by all nodes at the same time (because of the nature of the Controller Area Network bus ) and upon receiving this special message , all nodes will update the IDs of all nodes local messages . But A simple agreement protocol has two disadvantages . First of all , too much Controller Area Network bandwidth is wasted transmitting this special message every ' seconds . Moreover , a separate protocol must be run to elect a new leader in case the old leader fails . Instead , we use the following protocol which is not only robust but also consumes less bandwidth . Each node has a periodic timer which fires every ' seconds at which time the node takes the following actions : 1 . Set a flag to inform the Controller Area Network device driver that the identifier update protocol has begun . 2 . Configure the Controller Area Network network adapter to receive all messages (i.e. , enter promiscuous mode by adjusting the receive filter ) . 3 . Increment the data length (data length ) field of the highest-priority ready message on Each node . The first incremented-DL message to be sent on the Controller Area Network bus will serve as a signal to all nodes to update the IDs of all nodes messages . If the original data length of a message on the Controller Area Network bus is less than 8 , then incrementing the original data length of the message will result in transmission of one extra data byte (device drivers on receiving nodes strip this extra byte before forwarding a message on the Controller Area Network bus to the application as described later ) . If the original data length of the message data length is already 8 , Controller Area Network adapters allow the 4-bit data length field to be set to 9 (or higher ) but only 8 data bytes are transmitted . Now , each node starts receiving all messages transmitted on the Controller Area Network bus . The device driver on each node has a table listing the IDs of all message streams in the system along with the IDs of all message streams in the system data lengths . As messages arrive , the device driver compares messages data length field to the values in a table listing the IDs of all message streams in the system along with their data lengths until the device driver finds a message with an incremented data length field . All nodes receive a message on the Controller Area Network bus at the same time and All nodes take the following actions : 1 . Restore the receive filter to re-enable message filtering in the NA . 2 . If the local message whose data length field was incremented by the periodic timer has not been transmitted yet , then decrement the data length field back to the data length field original value . 3 . Update message IDs to reflect the new start of epoch . Each node receives the incremented-DL message at the same time , so the identifier update on each node starts at the same time . After the first incremented-DL message completes , the next-highest-priority message begins transmission . As long as All nodes complete their identifier updates before this message completes (a window of at least 55-s since a message on the Controller Area Network bus contains at least one data byte ) , all messages will have updated IDs by the time the next bus arbitration round begins and no priority inversion will occur . In case one or more nodes are slow and can not complete the identifier update within this window of time , one or more nodes can be configured to do the update while the n th message after the first incremented-DL message is in transmission , where n is a small number large enough to allow the slowest node to calculate all new IDs and then just write these to the NA while the n th message is in transmission . A simple agreement protocol incurs a network overhead of 16 bits every ' seconds (compared to 47 bits per epoch for the simple leader-based agreement protocol ) . Reception of the first incremented-DL message causes the device drivers to set the data length fields of their local messages back to their original values , but before this can complete , the next transmission (also with an incremented data length field ) has already started . These two messages have 8 extra data bits each (worst-case ) which leads to the 16-bit overhead . On the CPU side , the periodic process incurs some overhead . Moreover , while the network adapter 's filter is disabled , the device drivers must process two messages which may or may not be meant for that node . the device drivers must perform filtering in software and discard messages not meant for the device drivers node . In the following , we present a brief description of TouCAN , the problems faced when implementing real-time scheduling on Controller Area Network , and our solution to these problems for mixed traffic scheduler . TouCAN is a module developed by Motorola for on-chip inclusion in various microcontrollers . TouCAN lies on the same chip as the CPU and is interconnected to the CPU (and other on-chip modules ) through Motorola 's intermodule bus . Motorola is currently marketing the MC68376 [ 19 ] microcontroller which incorporates TouCAN with a CPU32 core . TouCAN has 16 message buffers . Each buffer can be configured to either transmit or receive messages . When more than one buffers have valid messages waiting for transmission , TouCAN picks the buffer with the highest-priority identifier and contends for the bus with this identifier . In this respect TouCAN differs from older Controller Area Network network adapters such as the Intel 82527 [ 20 ] which arbitrate between buffers using a fixed-priority , daisy-chain scheme which forces the host CPU to sort messages according to priority before placing messages in the network adapter buffers . This was one of the main reason we picked TouCAN for implementing mixed traffic scheduler . At this time , TouCAN is available only with the MC68376 microcontroller . To implement mixed traffic scheduler within EMERALDS on TouCAN , we would first have to port EMERALDS to the MC68376 microcontroller . To avoid this , we instead used device emulation [ 21 ] under which a general-purpose microcontroller is made to emulate a network adapter . This emulator interfaces to the host CPU through an I/O bus . This emulatorpresents the host CPU the same interface that the actual network adapter would . This emulator receives commands from the host CPU , performs the corresponding actions , and produces the same results that the actual network adapter would , thus providing accurate measurements of various overheads such as interrupt handling and message queuing on host CPU . we use a 68040 board to emulate the TouCAN module and connect the TouCAN module to the host CPU (another 68040 ) through a VME bus . 5.2 mixed traffic scheduler on Controller Area Network In implementing mixed traffic scheduler on Controller Area Network , we goal is to minimize the average overhead suffered by the host node for transmitting a message . the average overhead suffered by the host has the following components : 1 . Queuing/buffering messages in software if network adapter buffers are unavailable . 2 . Transferring messages to network adapter . 3 . Handling interrupts related to message transmission . In Controller Area Network , priority inversion can be unbounded . If the adapter buffers contain low-priority messages , these messages will not be sent as long as there are higher-priority messages anywhere else in the network . Con- sequently , a high-priority message can stay blocked in software for an indeterminate period of time , causing a high-priority message to miss a high-priority message deadline . Because of this priority inversion problem , any network scheduling implementation for Controller Area Network (regardless of which scheduling policy - deadline monotonic or mixed traffic scheduler - is being implemented ) has to ensure that adapter buffers always contain the highest-priority messages and only lower-priority messages are queued in software . Suppose B buffers are allocated for message transmission (usually B is about two-thirds of the total number of buffers ; see Section 6 ) . If the total number of outgoing message streams is B or less , then MTS's implementation is straight-forward : assign one buffer to each stream . Whenever the Controller Area Network device driver receives a message for transmission , the Controller Area Network device driver simply copies a message for transmission to the buffer reserved for that stream . In this case , no buffering is needed within the device driver which also means that there is no need for the Controller Area Network adapter to generate any interrupts upon completion of message transmission 3 , and this leads to the lowest-possible host CPU overhead . When number of message streams exceeds B , some messages have to be buffered in software . To reduce host CPU overhead , we want to buffer the fewest possible messages while avoiding priority inversion . Just as mixed traffic scheduler treats low-speed and high-speed messages differently for scheduling purposes , we treat these messages differently for implementation purposes as well . our goal is to keep the overhead for frequent messages (those belonging to high-speed periodic streams ) as low as possible to get a low average per-message overhead . In Our implementation , if the number of periodic high-speed message streams NHp is 3 The Controller Area Network adapter must be programmed to generate interrupts if messages are queued in software waiting for adapter buffers to become available , which is not the case here . less than B , then we reserve NHp buffers for high-speed periodic streams and treat NHp buffers for high-speed periodic streams the same as before (no buffering in software ) . The remaining buffers are used for high-speed sporadic , low-speed , and non-real-time messages . As these messages arrive at the device driver for transmission , these messages are inserted into a priority- sorted queue . To avoid priority inversion , the device driver must ensure that the L buffers always contain the L messages at the head of the queue . So , if a newly-arrived message has priority higher than the lowest-priority message in the buffer , a newly-arrived message `` preempts '' that message by overwriting a newly-arrived message . This preemption increases CPU overhead but is necessary to avoid priority inversion . a newly-arrived message stays in the device driver queue and is eventually transmitted according to a newly-arrived message priority . Among these L buffers , the buffer containing the I +1 th lowest priority message is configured to trigger an interrupt upon message transmission (I is defined later ) . This interrupt is used to refill the buffers with queued messages . I must be large enough to ensure that the bus does not become idle while the interrupt is handled and buffers are refilled . Usually an I of 1 or 2 is enough (which can keep the bus busy for 47-94 -s minimum ) . Note that this puts a restriction on L that this must be greater than I . Making L less than or equal to I can lead to the Controller Area Network bus becoming idle while the ISR executes , but makes more buffers available for high-speed periodic messages . This can be useful if low-speed messages make up only a small portion of the workload and high-speed sporadic messages are either non-existent or very few . If NHp - B then we must queue even high-speed periodic messages in software . Then we have a single priority-sorted queue for all outgoing messages and all B buffers are filled from this queue . For streams with dedicated buffers , the CPU overhead is just the calculation of the message identifier and transferring the message data and identifier to the network adapter . Note that message data can be copied directly from user space to the network adapter to keep overhead to a minimum . For messages which are queued in software , there is an extra overhead of inserting the message in the queue (including copying the 8 or fewer bytes of message data from user space to device driver space before inserting in the queue ) , plus the overhead for handling interrupts generated upon message transmission . This interrupt overhead is incurred once every I message transmissions , where Q is the number of buffers being filled from the queue (Q can be B or L depending on whether high-speed periodic messages are buffered or not ) . Also , each message will potentially have to preempt one other message . each message had already been copied to the network adapter once and now the network will have to be copied again , so the preemption overhead is equivalent to the overhead for transferring the message to the network adapter . If fixed-priority scheduling (such as deadline monotonic ) is used for Controller Area Network , some of these bits go unused . The idea behind mixed traffic scheduler is to use these extra bits to enhance network schedulability . mixed traffic scheduler places a quantized form of the message deadline in these extra bits while using the DM-priority of messages in the remaining bits . This enhances schedulability of the most frequent messages in the system (high-speed messages ) so that mixed traffic scheduler is able to feasibly schedule more workloads than deadline monotonic . Since message IDs are based on deadlines , message IDs must be periodically updated . We presented a protocol to perform this update without any priority inversion . a protocol consumes about 5-6 % of CPU time , but considering the large improvements in network schedulability that mixed traffic scheduler displays over deadline monotonic , this extra overhead is justified . We also presented a scheme to implement mixed traffic scheduler on the TouCAN network adapter which is representative of modern Controller Area Network network adapters . The biggest challenge in implementing Controller Area Network scheduling (be it mixed traffic scheduler or deadline monotonic ) is controlling priority inversion within the network adapter . We showed that because of CAN's characteristics (short message size ) , preemption of a message in the adapter by a newly-arrived , higher-priority outgoing message is an effective method for avoiding priority inversion . A future avenue of research can be to study message reception issues for Controller Area Network to try to reduce the average per-message reception overhead . Unlike message transmission , message reception does not depend on which network scheduling policy (deadline monotonic or mixed traffic scheduler ) is used .
